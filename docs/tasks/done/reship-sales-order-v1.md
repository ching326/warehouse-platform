# Reship a Sales Order v1

## Goal

Let an internal user pick a shipped sales order and reship it: create a new outbound
shipment for some or all of its lines, and record why the reship is needed
(missing, defect, other). The original sales order keeps its identity and status;
the reship is a new outbound shipment under the same sales order, and it flows through
the normal warehouse pick/pack/ship pipeline.

## Background

The data model already has most of the pieces; this task wires them together and closes
the gaps that stop a reship from being fulfilled.

- `OutboundOrder::REASON_RE_SHIP = 're_ship'` already exists. A reship is an outbound
  order whose `reason` is `re_ship`.
- `Issue` already models the reason: `issue_type` includes `missing`, `damaged`,
  `other`; it `belongsTo(SalesOrder)` and `belongsTo(OutboundOrder)`; it has a status
  workflow and a unique `issue_no` (`ISS-...`), created via an `ISS-PENDING-{uuid}`
  placeholder then `Issue::buildIssueNo($id)` (same pattern as `IssueCreate`).
- A sales order links to shipments through `outboundOrders()` (belongsToMany via
  `outbound_order_sales_order`) and `OutboundOrder.source_sales_order_id`.
- The outbound ref is generated by `OutboundOrder::buildRef($id, $tenantCode)`
  (`OB-{TENANT5}-{yymmdd}-{id4}`), also via an `OB-PENDING-{uuid}` placeholder first.

### Known constraints this spec must address

- The fulfillment pipeline filters `reason = REASON_CUSTOMER_ORDER` in many places
  (`FulfillmentIndex` x7, `FulfillmentPackService`, `FulfillmentPickSummary`,
  `FulfillmentPackStart`). A `re_ship` outbound reserves stock but, without changes,
  never appears in the warehouse workflow. See section 6.
- `reserveLine()` and the virtual-bundle line logic live as PRIVATE methods inside
  `OutboundOrderCreate`. A service cannot reuse them today. See section 3.
- `CourierExportService` does NOT filter by reason, but it relies on shipping fields;
  a reship must copy `shipping_method_id` so export does not block. See section 4.

## Two distinct axes (do not conflate)

- Why does this shipment exist? -> `OutboundOrder.reason = re_ship` (the action).
- Why are we reshipping? -> an `Issue` with `issue_type` missing / damaged / other.

## Decisions

- A reship creates a NEW `OutboundOrder` with `reason = re_ship`. Not a new sales order;
  it does not reuse or mutate the original shipment's outbound.
- Standard `OB-...` ref via `buildRef()`. Do not encode "reship" into the ref.
- The reship reason is an `Issue` (no new enum). UI exposes Missing (`missing`),
  Defect (`damaged`), Other (`other`). The Issue is created first, `status = open`,
  `outbound_order_id = original shipment` (where the problem occurred).
- Linkage (both directions):
  - `outbound_orders.reship_of_outbound_id` -> the original shipment (shipment lineage).
  - `outbound_orders.issue_id` -> the Issue this reship resolves (so two reships of the
    same original shipment each map to their own issue).
- Reships are partial-capable and selected by `sales_order_line_id` (not `sku_id`),
  because the same SKU can appear on multiple sales order lines.
- The reship outbound is created `status = reserved`: it reserves stock AND enters the
  fulfillment pipeline (see section 6). Stock shortage uses the same path as normal
  outbound creation (no new backorder logic).
- Warehouse: default to the ORIGINAL outbound's `warehouse_id`. If a different warehouse
  is chosen, it must reference an active warehouse (the `warehouses` table is global -
  no `tenant_id`). Reservation is scoped by `tenant_id` + `warehouse_id` +
  `stock_item_id`; insufficient stock there is handled by the normal reserveStock
  shortage error.
- Shipping: copy `shipping_method_id` and the unified shipping fields from the original
  outbound (so courier export does not block). Allow override later, not in v1.
- The original sales order's `order_status` and `fulfillment_status` are unchanged.
- Internal user only (`authorizeInternalUser()`).

## Explicit Non-Goals

- No new sales order; no change to sales order identity/uniqueness.
- No marketplace/external notification of the reship.
- No change to the Issue status workflow beyond creating it `open`.
- No backorder/partial-allocation logic beyond what outbound creation already does.
- Reships are NOT consolidated: `OutboundConsolidationService` stays
  `customer_order`-only and must not group reships.
- No tenant-initiated reship in v1.
- Do not touch `SalesOrder` or `InboundOrder` statuses.

## Eligibility

The Reship action is offered only when the sales order has at least one outbound in
shipped status. If exactly one shipped outbound exists, auto-select it as the original;
if more than one, the user picks which shipment had the problem.

## 3. Service

New file: `app/Services/Outbound/ReshipSalesOrderService.php`

```php
class ReshipSalesOrderService
{
    /**
     * @param  array<int, array{sales_order_line_id:int, qty:int}>  $lines  (qty > 0)
     */
    public function reship(
        SalesOrder $salesOrder,
        OutboundOrder $originalOutbound,
        ?int $warehouseId,          // null => use original outbound warehouse
        string $issueType,
        array $lines,
        ?string $note,
    ): OutboundOrder;
}
```

Behavior (single DB transaction):

1. Validate:
   - `$salesOrder` and `$originalOutbound` share the tenant; the outbound is linked to
     this sales order (pivot or `source_sales_order_id`) and is shipped.
   - `$issueType` is in `reshipReasonOptions()` keys.
   - Each `sales_order_line_id` belongs to `$salesOrder`; `qty >= 1` and
     `qty <= salesOrderLine.quantity`.
   - Resolve warehouse: `$warehouseId ?? $originalOutbound->warehouse_id`; if provided,
     it must reference an active warehouse (warehouses are global; no `tenant_id`).
     Reservation uses `tenant_id` + `warehouse_id` + `stock_item_id`; a shortage there
     surfaces via the normal reserveStock error.
2. Create the Issue first (so the reship can reference it):
   - `issue_no => 'ISS-PENDING-'.Str::uuid()` placeholder, then after insert
     `update(['issue_no' => Issue::buildIssueNo($issue->id)])`.
   - `tenant_id`, `sales_order_id => $salesOrder->id`,
     `outbound_order_id => $originalOutbound->id` (where the problem happened),
     `issue_type => $issueType`, `status => Issue::STATUS_OPEN`,
     `reported_at => now()`, `note`, `created_by_user_id => auth id`.
   - This Issue is shipment-level and has NO issue lines. `IssueCreate` normally
     requires lines for an order-linked issue; this directly-created reship Issue is a
     deliberate exception, so the Issue detail view must tolerate an empty issue-line
     set (render cleanly with zero lines).
3. Create the reship outbound:
   - `reason => REASON_RE_SHIP`, `status => STATUS_RESERVED`
   - `source_sales_order_id => $salesOrder->id`
   - `reship_of_outbound_id => $originalOutbound->id`
   - `issue_id => $issue->id`
   - `tenant_id`, `warehouse_id` (resolved)
   - copy recipient fields, `shipping_method_id`, and the unified shipping fields from
     `$originalOutbound`
   - `ref => 'OB-PENDING-'.Str::uuid()` placeholder
   - link the sales order via the pivot.
4. For each selected line: load the `SalesOrderLine` (tenant-scoped), resolve its SKU,
   and call the shared `OutboundLineBuilder` with `reserve = true`. (Bundles handled by
   the builder.)
5. Finalize ref: `update(['ref' => OutboundOrder::buildRef($outbound->id, $salesOrder->tenant->code)])`.
6. Return the reship outbound.

```php
public static function reshipReasonOptions(): array
{
    return [
        Issue::TYPE_MISSING => __('outbound.reship_reason_missing'),
        Issue::TYPE_DAMAGED => __('outbound.reship_reason_defect'),
        Issue::TYPE_OTHER   => __('outbound.reship_reason_other'),
    ];
}
```

## 6. Make reships flow through the warehouse pipeline

Add a helper and use it where the pipeline currently hardcodes `customer_order`:

```php
// OutboundOrder
public static function fulfillableReasons(): array
{
    return [self::REASON_CUSTOMER_ORDER, self::REASON_RE_SHIP];
}
```

Replace `->where('reason', REASON_CUSTOMER_ORDER)` with
`->whereIn('reason', OutboundOrder::fulfillableReasons())` in BOTH the queue readers AND
the per-outbound row actions, so warehouse staff can hold / release / remap / pack /
ship a reship like any shipment:

- `app/Livewire/FulfillmentIndex.php` - ALL seven reason filters (verified): the queue reader
  `render()`, plus the row actions `remapShipping()`, `holdSelected()`,
  `releaseHoldSelected()`, `packingStartedRefs()`, `holdOutboundIds()`, and
  `selectedOutboundOrderIds()`. None of these is a consolidation/combine query.
- `FulfillmentPackService`
- `FulfillmentPickSummary`
- `app/Livewire/FulfillmentPackStart.php`

Do NOT change (these stay `customer_order`-only):

- `OutboundConsolidationService` - reships are standalone and must not be consolidated
  or joined into ship-together groups.
- The Sales Order page customer actions (`customerOutboundOrders()` /
  `customerOutboundFor()` and the SO-page hold / split / rebuild) - section 5.

`CourierExportService` has no reason filter, so no change there; correctness depends on
the reship carrying `shipping_method_id` (section 3).

Reships therefore appear in the queue as their own standalone outbound (single sales
order, no ship-together grouping) and can be picked, packed, held, released, remapped,
and shipped normally.

## Notes

- The reship's identity is its own outbound `id` / `OB-...` ref. The sales order keeps
  its identity (internal `id`; `platform_order_id` is display-only and not unique).
- `reship_of_outbound_id` is shipment lineage; `issue_id` is the reason record. They are
  intentionally separate (lineage is derivable via `issue.outbound_order_id`, but the
  direct link is cheaper for display and for "reships of this shipment").
- Issue is order/shipment level, not line level; per-line detail can go in the note.
